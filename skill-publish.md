# Playbook: Skill Publish

> **类型：** How-to Guide / 可复用发布执行手册
> **最后更新：** 2026-04-26
> **维护人：** 范蠡（流程框架）+ 万三（调度约束）

---

## 1. 适用场景

- 新 Skill 首次公开发布到 GitHub
- 已发布 Skill 的版本迭代（patch / minor / major）
- 需要跨 runtime 一起交付的 Skill
- 需要同时覆盖代码、文档、安装、卸载、验证、发布页
- 需要把设计审查、实现、双轨验收、发版动作串成一条闭环
- 适用于 OpenClaw / Claude Code / Opencode / Codex 这类多运行时兼容 Skill
- 适用于“先内部打磨，后对外发 alpha / beta / rc / stable”的节奏
- 适用于已有仓库、已有脚手架、准备进入公开可安装状态的项目

## 2. 不适用场景

- 仅自己本地使用、不准备公开的 Skill
- 只有设计文档，还没到代码可运行阶段
- 只有原型脚本，没有版本号、没有仓库、没有测试基线
- 只改了一两句文案，不需要形成 release 的小变更
- 只是更新内部知识库，不需要外部仓库或 npm 分发
- 项目还没做清理，`dist/`、`node_modules/`、敏感信息风险未收敛

## 3. 前置条件

### 3.1 凭证

- GitHub 凭证可用
- 如果要发 npm，npm 凭证可用
- 凭证类问题先查记忆，再决定是否让用户补操作
- GitHub PAT 参考记忆中的已知路径：`~/code_key`
- 发版前先确认 `gh auth status` 可用
- 若涉及 npm publish，需确认 npm token 或已登录状态
- 如果需要额外平台权限，必须在发布前一次性确认，不要到 Step 5 才发现鉴权缺失

### 3.2 工具

- `git`
- `gh`
- `node`（版本需符合 Skill 的 runtime 要求）
- 对应包管理器（`pnpm` / `npm`）
- 项目自己的测试命令
- 项目自己的安装 / 卸载脚本
- Shell 环境可执行 setup / uninstall / e2e 脚本

### 3.3 仓库与脚手架

- 已有 Skill 仓库
- 仓库已有基本目录结构
- `SKILL.md` 已存在
- 关键 references 文档已存在或计划补齐
- `package.json`、版本号、README 至少有最小可维护状态
- `.gitignore` 已覆盖构建产物、依赖目录、缓存目录
- 若支持多 runtime，适配层或集成说明已成形

### 3.4 路径

- Playbook 模板：`~/open-claw-output/doc/playbooks/playbook-template.md`
- GitHub Release Playbook：`~/open-claw-output/doc/playbooks/github-release.md`
- 本 Playbook 产出路径：`~/open-claw-output/doc/playbooks/skill-publish.md`
- 技术产出目录：项目仓库根目录
- 审查报告目录：`~/open-claw-output/doc/`
- 发布相关文档建议统一放在仓库根目录或 `docs/`

### 3.5 团队角色

#### 必需角色

- **万三**：调度、拍板、推进顺序、验收门槛、对外发布口径
- **鲁班**：主实现，修 Blocker，补脚本，补文档落地
- **包拯**：代码视角审查，grep + 测试 + 元信息核验
- **商鞅**：用户视角审查，mktemp 沙箱 + 端到端 + 文档对照
- **纪昀**：里程碑日记落盘、发布后记忆更新

#### 条件必需角色

- **图灵**：当项目仍处于方案未收口、设计需要重写或结构升级时必需
- **移公**：当发布涉及 shell / 环境 / 安装链路 / npm / gh / CI / 权限细节时建议加入

#### 可选角色

- 张仪：要同步社区内容、官网文案、传播稿时加入
- 西施：要补视觉资产、徽章、截图、头图时加入

### 3.6 开工前最小门槛

- 方案不是“脑内共识”，而是已有文档版本
- 方案已经过至少一轮独立审查
- 版本号升级策略已拍板
- 是否 prerelease 已拍板
- 是否包含 npm publish 已拍板
- 最小测试基线已定义
- 文档最小交付面已定义
- 发布是否需要支持 4 runtime 已拍板

## 4. 角色分工

### 4.1 万三（主会话）

- 决定是否进入发布流
- 决定版本号等级
- 决定 alpha / beta / rc / stable 标记
- 决定是否并行拆 ticket
- 决定每阶段的完成门槛
- 决定是否允许带 Suggestion 发布
- 决定 Blocker 是否已实质闭合
- 决定是否延后发布而不是强行压线
- 处理“继续修”还是“先发再补”的最终取舍
- 负责串联本 Playbook 与外部 Playbook
- 负责提醒：凭证问题先查记忆，别让用户重复登录

### 4.2 图灵（方案设计）

- 起草设计文档
- 吸收显超修订规约
- 生成 v1.0 → v1.1 等结构性迭代
- 在设计阶段把 ticket 边界、DoD、测试面、兼容面说清楚
- 输出必须能被后续 ticket 化，而不是只有大段叙述
- 如涉及元信息交叉核验，优先使用更稳的模型链路

### 4.3 鲁班（主实现）

- 依据 ticket 实装
- 每个 ticket 自带 DoD
- 修审查发现的问题
- 补自动化测试或配合商鞅落地测试入口
- 发版前补 README / QUICKSTART / package 元信息等落地项
- 输出 commit 与验证证据

### 4.4 包拯（代码视角审查）

- 检查实现与设计是否一致
- 检查 grep 残留
- 检查测试是否真的跑过
- 检查 CHANGELOG / RELEASE NOTES / README / 版本号等元信息一致性
- 检查“声明已修复”的条目能否在正文定位
- 输出 Blocker / Major / Suggestion / Nit 分层结果
- 必须诚实，不能过报

### 4.5 商鞅（用户视角审查）

- 在 `mktemp` 沙箱做真实安装与卸载
- 从用户入口出发做端到端校验
- 对照文档验证实际行为
- 检查安装、使用、卸载、目录语义、参数一致性
- 抓包拯不容易看到的真实体验问题
- 输出 PASS / FAIL 与明确证据

### 4.6 移公（环境与发布技术）

- 处理 shell、gh、npm、安装器、权限、路径、CI、网络、发布命令
- 补深 GitHub / npm 的技术细节
- 当主问题已经不是产品逻辑，而是环境与工具链问题时介入

### 4.7 纪昀（记忆与里程碑）

- 发版后写日记
- 更新里程碑
- 协助同步 MEMORY.md / TODO.md
- 不参与发布决策，但负责把事实写下来，避免下次失忆

## 5. 关键决策点

### 5.1 何时升 patch / minor / major

| 版本类型 | 何时使用 | 例子 |
|---|---|---|
| `v1.0.1` patch | 修 bug、补文档、补测试、防回归、无破坏性变更 | uninstall 修复、README 勘误 |
| `v1.1.0` minor | 新增能力、扩新 runtime、补新命令、兼容升级 | 新增 Codex 支持、引入 mirror 流程 |
| `v2.0.0` major | 设计模型/目录结构/接口语义/配置协议破坏性变化 | 改配置 schema、迁移存储形态 |

**判断规则：**

- 用户已有安装若无需改使用方式，优先 patch / minor
- 文档要重写但使用方式不变，不自动等于 major
- 需要迁移脚本、目录迁移、配置迁移、兼容策略变化，优先 major
- 如果团队拿不准，宁可先保守为 prerelease minor，再收集反馈

### 5.2 何时打 prerelease

| 标记 | 适用条件 |
|---|---|
| `alpha` | 功能主链已通，但仍在验证真实可用性 |
| `beta` | 功能主链稳定，边角兼容和文档仍在打磨 |
| `rc` | 预计无结构性改动，只剩最后确认 |
| stable | 双轨审查长期稳定，全量门槛满足 |

**alpha 适用信号：**

- 首次公开
- 刚做完真实环境闭环
- 测试已经全绿，但生态面尚未充分验证
- 安装链路刚跑通

**不要过早 stable：**

- 若这是第一次真实安装给外部用户用
- 若卸载、升级、兼容路径还没经过商鞅实跑
- 若 release notes 仍在补洞

### 5.3 测试基线红线

发布前必须定义“什么叫全绿”。

至少包括：

- unit tests
- setup / install tests
- e2e tests
- uninstall / teardown tests
- 防回归测试
- 若有 CLI，则含核心命令 smoke tests

**最低规则：**

- 不是“挑几条绿了就发”，而是约定基线里的项目必须全绿
- 任一基线红 = 不发
- flake 不等于可忽略，flake 是待修问题
- 新修一个 Blocker，至少补一条对应防回归

**覆盖率建议：**

- 这里不强制单一百分比阈值
- 但关键链路必须有直接覆盖
- 安装 / 卸载 / 主命令 / 目录决策 / 配置解析不得裸奔

### 5.4 审查通过标准

至少满足：

- 包拯：`0 Blocker`
- 商鞅：端到端 `100% PASS`
- 万三：确认 ticket DoD 都已闭合
- 文档与实现无关键矛盾
- 发布元信息一致

**注意：**

- 包拯通过 ≠ 可以直接发
- 商鞅通过 ≠ 不用看代码质量
- 双轨都通过，才进入发布执行

### 5.5 并行度上限

- 实装建议并行 ≤ 3
- 超过 3 条并行容易出现 git 冲突、信息噪音、验收压力上升
- 同一阶段并行要按边界拆，不要多人碰同一目录同一文件
- 如果必须并行，先写清楚 ticket 归属与合并顺序

### 5.6 发布前发现遗漏时怎么办

优先判断遗漏类型：

- **Blocker 级**：推迟发布，先修
- **Major 级但不破坏主链**：看是否能快速补齐；若不能，降为 prerelease 或推迟
- **文案 / 截图 / 建议项**：可发后补，但必须入 NOTES 或 TODO

**判断标准：**

- 是否影响首次安装成功
- 是否影响卸载完整性
- 是否影响版本信用
- 是否会让 release notes 说的话对不上真实行为

## 7. 步骤

## Phase 1：方案设计与审核

### Step 1.1 起草设计文档

**目标**

- 把 Skill 要解决的问题、边界、目录、命令、测试面、兼容面一次讲清楚
- 让后续能拆 ticket，而不是只留下抽象想法

**动作**

- 图灵起草 v1.0
- 写清楚适配哪些 runtime
- 写清楚安装 / 卸载 / 配置 / 目录 / 失败模式
- 写清楚测试与验收标准
- 把设计文档版本化保存

**预期输出**

- `xxx-design-v1.0.md`
- 若有变更历史，补 `CHANGELOG`
- 可定位的章节结构

**异常情况**

- 文档只会讲概念，不可 ticket 化 → 退回补结构
- 文档没有把目录语义写清楚 → 退回补路径与默认值
- 文档没有测试面 → 退回补 DoD 与验证基线

### Step 1.2 显超出修订规约

**目标**

- 把“哪里不行、怎么改、优先级是什么”一次性压成明确规约

**动作**

- 万三根据显超反馈整理修订清单
- 规约按 Blocker / Major / Suggestion 分层
- 规约尽量精确到章节、术语、结构、冲突点

**预期输出**

- 修订规约文档
- 带编号的修订点（如 R1、B1、M1）

**异常情况**

- 规约太模糊，图灵无法对号入座 → 重写为编号清单
- 同一个问题重复出现在多个条目 → 先去重再派工

### Step 1.3 图灵 v1.0 → v1.1 修订

**目标**

- 按规约做结构性闭合，而不是表面改字

**动作**

- 图灵逐条落实规约
- 返回时必须附自查证据
- 自查至少包括：grep 清查、章节对照、变更位置说明

**预期输出**

- `xxx-v1.1.md`
- 逐条修订说明
- 关键术语残留检查结果

**异常情况**

- 只在回复里说修了，正文对不上 → 直接退回
- CHANGELOG 说全闭合，但正文仍有残留 → 直接退回
- 版本覆盖旧文档 → 违规，必须恢复版本化轨迹

### Step 1.4 包拯独立审查

**目标**

- 从设计质量与元信息一致性的角度做独立可信审查

**动作**

- 包拯不参与起草，只做审查
- 检查修订点是否真的闭合
- 检查 grep 残留
- 检查 CHANGELOG 与正文一致性
- 检查关键表述是否双轨冲突

**预期输出**

- 独立审查报告
- Blocker / Major / Suggestion 分层

**异常情况**

- 包拯连续两轮过报 → 按规则换角色或换模型
- 包拯只给“总体没问题”，不给证据 → 退回补证据

### Step 1.5 万三确认 0 Blocker

**目标**

- 把设计阶段正式收口，进入可实现状态

**动作**

- 万三抽查关键 grep
- 万三核对 0 Blocker 是否成立
- 确认是否可以开始 ticket 化与实装

**完成标准**

- 包拯 0 Blocker
- 万三抽查通过
- 文档版本冻结为本轮实现依据

**阶段出口**

- 进入 Phase 2

## Phase 2：实装

### Step 2.1 万三拆 ticket

**目标**

- 把设计文档变成可并行、可验收、可回滚的实现任务

**动作**

- 按依赖关系拆 ticket
- 建议并发 ≤ 3
- 每个 ticket 写清楚范围、DoD、边界、工时上限
- 优先拆成“能验收的完整小块”，不要拆成机械零件

**参考范式**

- self-learning-loop v1.1 三件套并行：T-SLL-005 / 006 / 012

**预期输出**

- ticket 清单
- 每个 ticket 的 DoD
- 依赖顺序与并行关系

**异常情况**

- ticket 太大，无法一轮验收 → 再拆
- ticket 相互重叠，同碰一个文件 → 重划边界
- ticket 没 DoD → 不准派发

### Step 2.2 鲁班并行实装

**目标**

- 在不牺牲可审查性的前提下加速实现

**动作**

- 鲁班按 ticket 实装
- 每个 ticket 返回 commit、测试、变更摘要
- 对并行任务，尽量按目录或功能边界隔离

**并行硬规则**

- 推荐 ≤ 3 并发
- 同一文件不要双人同改
- 合并前看 `git status`
- 避免 `git add -A` 把别人工作顺手带进来

**预期输出**

- 代码提交
- 测试通过证据
- 对应 DoD 闭合说明

**异常情况**

- 发现现有代码已部分实现 → 先审计，再改 task，避免重写浪费
- 子代理执行中断但代码已落地 → 万三及时抢救 commit
- 并发污染到他人文件 → 立刻拆分提交并补规则

### Step 2.3 商鞅写自动化测试

**目标**

- 在实现阶段就把关键主链守住，而不是等发布前临时补

**动作**

- 给每个高风险 ticket 补对应测试
- 尤其补 install / uninstall / config / e2e / regression
- 对用户真实操作链路，优先做 e2e

**预期输出**

- 自动化测试文件
- 测试入口命令
- 关键断言列表

**异常情况**

- 只有 unit，没有 setup/e2e → 视为发布面未闭合
- 测试只能在开发机跑，沙箱跑不了 → 补环境隔离

### Step 2.4 每个 ticket 做 DoD 闭合

**目标**

- 保证每张票都能独立说清楚“完成了什么”

**DoD 最小模板**

- 功能可用
- 测试已补
- 文档有同步
- 边界行为已检查
- 没有新的已知 Blocker

**完成标准**

- ticket 有代码
- 有验证
- 有闭合说明
- 能被下一阶段审查复现

**阶段出口**

- 所有进入发版范围的 ticket 均已闭合

---

## Phase 3：双轨审查

### Step 3.1 包拯：代码视角审查

**目标**

- 从实现质量、元信息一致性、已知风险残留三方面做独立审查

**动作**

- 跑 grep
- 跑关键测试
- 核对 README / package / CHANGELOG / RELEASE NOTES / 版本号
- 检查“说修了”和“真的修了”是否一致

**重点检查项**

- 旧术语是否残留
- 旧目录语义是否残留
- 测试数字是否与真实结果一致
- README 是否引用不存在文件
- package 版本与 tag 计划是否一致

**预期输出**

- 包拯审查报告
- Blocker / Major / Suggestion / Nit

### Step 3.2 商鞅：用户视角审查

**目标**

- 站在第一次安装和第一次使用的人视角，验证“文档真能带人走通”

**动作**

- 建 `mktemp` 沙箱
- 执行安装
- 运行最核心命令
- 检查目录落点
- 跑卸载
- 对照 README / QUICKSTART / references

**重点检查项**

- 选项名是否一致
- 卸载是否真的干净
- 默认目录语义是否一致
- 文档说的路径和实际路径是否一致
- 从任意 cwd 是否还能正常工作

**预期输出**

- PASS / FAIL
- 失败复现步骤
- 证据截图或日志片段

### Step 3.3 任一发现 Blocker → 鲁班修复

**目标**

- 审查不是终点，修复闭环才是

**动作**

- 由鲁班修复 Blocker
- 每修一个问题，补一条对应防回归
- 更新文档或脚本

**预期输出**

- 修复 commit
- 新增测试
- 问题闭合说明

### Step 3.4 双轨复审

**目标**

- 防止“修了一个又漏另一个”

**动作**

- 包拯复审代码/元信息面
- 商鞅复跑用户链路
- 万三确认不是“单轨假绿”

**完成标准**

- 包拯 0 Blocker
- 商鞅 100% PASS
- Blocker 相关新增测试已绿

**阶段出口**

- 进入 Phase 4

## Phase 4：发版准备

### Step 4.1 写 RELEASE NOTES

**目标**

- 对外清楚说明：这版发了什么、修了什么、基线是什么、有哪些已知边界

**必须覆盖**

- ticket 范围
- Blocker 修复概览
- 测试基线
- 版本定位（alpha / beta / rc / stable）
- 是否有破坏性变化
- 如有已知限制，要诚实写明

**不要写成**

- 空泛营销文案
- 与正文对不上的“全部闭合”大话
- 不可验证的夸张结论

### Step 4.2 测试基线全绿

**目标**

- 在发版动作前做最后一次全链路确认

**至少检查**

- unit
- setup
- e2e
- uninstall / teardown
- 防回归
- 若项目定义了 `verify`，优先统一走 `verify`

**通过标准**

- 本轮定义的基线必须全绿
- 不能只跑一部分后宣称发布就绪

### Step 4.3 文档同步

**目标**

- 让用户拿到 release 后，文档能真实带路

**至少同步清单**

- README
- QUICKSTART
- 关键 references 文档
- SKILL.md 中对外可见描述
- 如有 runtime integration 文档，也需同步

**重点检查**

- 安装命令
- 目录说明
- 参数名
- 默认值
- 版本号
- 徽章数据
- 文件树引用

### Step 4.4 更新版本号 / 徽章 / package.json

**目标**

- 保证仓库元信息与发布动作一致

**检查项**

- `package.json` version
- README 徽章
- 文档中的测试数字
- release title
- tag 名称

**完成标准**

- 不存在“代码 v1.1.0，README 还写 1.0.x”这类错位

### Step 4.5 发布前总复核

**目标**

- 把真正昂贵的错误挡在 push/tag 之前

**总复核问题**

- 现在发出去，第一次安装的人会不会直接踩坑？
- release notes 有没有一句过报？
- 文档说的话能不能被商鞅复现？
- 是否还留着不该入库的产物？
- 当前版本定位是不是应该继续挂 prerelease？

**阶段出口**

- 进入 Phase 5

## 8. 卡点 & 兜底（决策树）

| 卡点 | 第一兜底 | 第二兜底 |
|---|---|---|
| 包拯过报一次 | 退回重审，要求证据化 | 万三抽查后删不实条目 |
| 包拯连续两轮过报 | 触发换角色/换模型规则 | 改由万三或其他审查链收尾 |
| 商鞅发现 Blocker | 鲁班修复 + 补防回归 | 双轨复审，不许单轨放行 |
| 实装并行冲突 | rebase / 拆提交 / 明确文件边界 | 降并发到 ≤ 2 |
| `git add -A` 混入他人文件 | 拆 commit 重整 | 后续并行加文件边界规则 |
| 测试 flaky | 先定位是否隔离问题 | ratchet 模式：每修一个补一条防回归 |
| 沙箱绿、开发机红 | 在真实环境复核 | 补 workspace/path 隔离逻辑 |
| 文档与实现不一致 | 先修实现还是修文档，必须选一个收口 | 重跑商鞅文档对照验收 |
| gpt-5.4 schema reject | 先确认是不是工具 schema 注入问题 | 切 `opus-4.6` 兜底 |
| 发布前发现遗漏 | Blocker 就延后 | Major 视影响降 prerelease 或补修 |
| GitHub 凭证卡住 | 先查记忆和 `~/code_key` | 再走 `gh auth login` |
| npm 发布卡住 | 先暂停 npm，不拖 GitHub 主发布 | 走 `npm-publish.md` 兜底 |

### 8.1 包拯过报怎么办

- 一次过报：本轮作废，退回重审
- 连续两轮过报：触发换角色或换模型规则
- 不能把“回复里解释过”当成正文已经修复
- CHANGELOG 对不上正文的条目直接删

### 8.2 商鞅发现 Blocker 怎么办

- 不争论体验问题“小不小”
- 先按 Blocker 处理
- 由鲁班修
- 修后必须补防回归
- 再做包拯 + 商鞅双轨复审

### 8.3 实装并行冲突怎么办

- 先停手，不要硬 merge
- 明确各 ticket 文件边界
- 能 rebase 就 rebase
- 需要时拆成多个 commit
- 实在冲突高，就降并发

### 8.4 测试 flaky 怎么办

- 先判断是否环境污染
- 再判断是否时间竞争 / 路径冲突 / 沙箱假绿
- 采用 ratchet 模式：每修一个问题，加一个防回归
- 不接受“偶发红先忽略”作为发布前标准

### 8.5 gpt-5.4 schema reject 怎么办

- 先确认是不是模型本身挂了
- 再确认是不是工具 schema 注入导致 strict 校验失败
- 若是该组合失稳，切 `opus-4.6`
- 不要把“一个组合故障”误判成“整个模型不可用”

### 8.6 发版前发现遗漏怎么办

先分级：

- 安装/卸载/目录语义/参数不一致 → Blocker，延后
- README 数字未更新、次级文档漏一处 → 看影响，能补就补
- 非关键建议项 → 发布后补，但记入 TODO / notes

原则：

- 信用比赶时间重要
- 该推迟就推迟
- 不做“明知有坑还照发”的漂亮仗

## 10. 历史教训

### 2026-04-25 alpha.1：`dist/` + `node_modules/` 入库翻车

- **发生了什么**：发布准备阶段把不该入库的构建产物和依赖目录带进仓库
- **暴露问题**：`.gitignore` 覆盖不足；发布前没有做充分清仓检查
- **教训**：发版前必须做入库污染检查
- **后续固化**：补 `.gitignore`；补 uninstall-e2e；以 ratchet 模式增加防回归
- **影响阶段**：Phase 4、Phase 5

### 2026-04-26 alpha.2：包拯审完，商鞅仍发现 3 Blocker（B4/B5/B6）

- **发生了什么**：代码视角已认为接近可发，但商鞅在真实用户链路里抓出 uninstall 选项不一致、卸载不完整、目录语义混乱等问题
- **暴露问题**：单靠代码视角审查不够
- **教训**：双轨审查不可替代
- **后续固化**：包拯 + 商鞅 双绿灯才允许发版
- **影响阶段**：Phase 3

### 2026-04-19 v5.0.x：连续两轮 CHANGELOG 过报

- **发生了什么**：CHANGELOG 声称“全部闭合”，正文却仍有残留或元信息对不上
- **暴露问题**：修订者把“回复里解释过”当成“正文已修复”
- **教训**：CHANGELOG 必须能映射正文；连续两轮过报就换角色/换模型
- **后续固化**：AGENTS.md 引入“信用比速度重要”“连续两轮过报换角色”规则
- **影响阶段**：Phase 1、Phase 3、Phase 4

### 2026-04-22 Copilot schema reject：gpt-5.4 失稳时切 opus-4.6

- **发生了什么**：gpt-5.4 + 工具注入组合出现 schema reject，导致子代理无法稳定执行
- **暴露问题**：不能把组合级故障误判成模型整体失效
- **教训**：发布链路必须有模型兜底；架构/审查任务要有备用模型链
- **后续固化**：gpt-5.4 失稳时切 `opus-4.6`
- **影响阶段**：Phase 1、Phase 3

### 2026-04-25：subagents steer 实际是 restart，不是追加

- **发生了什么**：执行中追加指令，结果把子代理重启，前面几分钟工作白做
- **暴露问题**：派单不完整，且把 steer 当成 message append
- **教训**：派单尽量一次写全；要追加需求，优先新开子任务
- **影响阶段**：Phase 2

### 2026-04-25：沙箱绿、真实环境红

- **发生了什么**：子代理在干净 home 里报告测试通过，但开发机真实环境因 audit 数据污染导致 11 项失败
- **暴露问题**：环境隔离不足
- **教训**：发布前必须在真实环境再复核一次；不能只信沙箱
- **后续固化**：加 workspace resolver、隔离变量、端到端复核
- **影响阶段**：Phase 2、Phase 3、Phase 4

### 2026-04-26：凭证问题先查记忆，不要先让用户登录

- **发生了什么**：发布时先让显超手动 `gh auth login`，后面才查到记忆里已写明 `~/code_key`
- **暴露问题**：违反“能多走半步”与启动检索规则
- **教训**：凭证类问题先查记忆、文件、历史路径
- **影响阶段**：Phase 5
